# 代码注释规范

## 为什么要规范

烂注释比没注释更可怕——它会误导人，让你花半小时跟踪一条已经不存在的逻辑。

## 注释层级

### 🔵 文件头注释（必须）
写这个文件是干什么的，不要写版本号和修改历史（Git 已经记了）。

```js
// ✅ 好 — 麒麟菜单管理服务
// 统一管理分类（MenuCategory）和菜品（MenuItem）

// ❌ 差 — 菜单管理服务 v2.1（最后修改: 2026-04-20 by 某人）
//     版本号和修改记录让 Git 管，别写代码里
```

### 🟢 注释代码逻辑（尽量少写）
好代码应该像写文章一样自解释，注释只讲代码讲不清楚的事情。

```js
// ✅ 好 — 解释"为什么"，而不是"是什么"
// slug 在 URL 中会被直接使用，中文 slug 会变成编码字符不友好
// 所以只保留小写英文和数字
const baseSlug = name.replace(/[\u4e00-\u9fa5]+/g, '').toLowerCase()

// ❌ 差 — 代码已经说了"生成唯一slug"
// 生成唯一的slug
const slug = await this.generateUniqueSlug(storeData.name, tenantId)

// ❌ 差 — 毫无信息量的重复
// 创建店铺
const store = await this.db.store.create({...})
```

### 🟡 修复记录（谨慎使用）
修复 bug 后不要删掉修复痕迹来冒充一次写对的，但不要留太长。

```js
// ✅ 好 — 来源清晰，解释为什么特殊处理
// 旧数据中有中文 slug（如"水岸东方"），查询前做规范化
const safeSlug = slug.replace(/[\u4e00-\u9fa5]+/g, 'store-')

// ❌ 差 — 已经过时的修复记录
// 之前一直报500，因为createError没有statusCode（2026-04-24已修复）
// 现在改了error.service.js加上statusCode就行了
```

### 🔴 临时标记（必须带日期和计划）
临时标记一定要有"什么时候加的 + 打算什么时候解决"。

```js
// TODO [2026-04-24]: 这个接口支持分页后再给前端 true
// 计划: 等 潘哥 确认分页参数后改成动态值
return { page: 1, total: 100 }
```

## 三个"必须"

1. **改代码必须改注释** — 不改注释的改动是不完整的改动
2. **废弃代码必须删** — 不要注释掉代码留着"以后可能用"。Git 里有历史，需要的时候 checkout
3. **TODO 必须有清理计划** — 如果写 "TODO" 但没有日期和负责人，等于没写

## 注释模板

### 文件注释
```
// XXX项目 - 模块名
// 一句话说明这个文件的职责
```

### 函数注释（需要时才写）
```js
/**
 * 解释为什么要这么设计
 * 不要复述代码干了什么
 * @param {string} xxx - 只说类型和含义，不说"xxx是..."
 */
```

### 特殊逻辑注释
```js
// ⚠️ 小心: [具体原因]
// 例如: 这个API的storeId参数是slug，后端会做转换
```

## 清理原则

**当看到一个过时的注释：**
1. 先确认代码逻辑是否还对
2. 注释对不上就删注释（不要改注释去迎合代码——如果代码想改，先和 潘哥 确认）
3. 注释和代码都对不上——可能是 bug，需要修复

**当看到一个被注释掉的代码块：**
1. 看 Git blame 知道是谁、什么时候注释的
2. 如果超过一周，直接删
3. 如果不到一周，改成 TODO 带上日期
