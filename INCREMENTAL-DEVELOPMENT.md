### **阶段 1: 核心MVP - 基础认证系统 (当前阶段)**
**目标**: 跑通最基本的用户注册、登录、JWT认证流程

#### **功能范围**:
1. **用户注册** (邮箱/密码)
2. **用户登录** (邮箱/密码 + JWT)
3. **JWT Token管理** (签发、验证、刷新)
4. **基础用户信息** (用户表、Profile)
5. **密码安全** (bcrypt加密、密码策略)

#### **技术实现**:
- 后端: Fastify + JWT + bcrypt
- 前端: React + React Router + Axios
- 数据库: PostgreSQL + Prisma
- 认证: JWT + HTTP-only cookies

#### **验收标准**:
- ✅ 用户可以注册新账号
- ✅ 用户可以登录获取JWT Token
- ✅ Token可以用于API认证
- ✅ 密码安全存储 (bcrypt)
- ✅ 所有配置通过环境变量管理
- ✅ 所有API路由规范化管理

## 🔧 渐进开发工作流

### **每个阶段的开发流程**:
```
1. 需求分析 → 2. 技术设计 → 3. 代码实现 → 
4. 单元测试 → 5. 集成测试 → 6. 代码审查 → 
7. 部署测试 → 8. 用户验收 → 9. 文档更新
```

### **代码提交规范**:
```bash
# 功能提交
git commit -m "feat(auth): 用户注册功能实现"

# 修复提交  
git commit -m "fix(config): 修复环境变量加载问题"

# 文档提交
git commit -m "docs(api): 更新API文档"

# 重构提交
git commit -m "refactor(routes): 重构API路由管理"

# 测试提交
git commit -m "test(auth): 添加用户认证测试"
```

### **分支管理策略**:
```
main (生产就绪)
  ↑
develop (开发集成)
  ↑
feature/* (功能分支)
  ↑
hotfix/* (紧急修复)
```

## 🛡️ 硬编码防护机制

### **防护层级**:
1. **环境变量层**: 所有配置必须通过环境变量
2. **配置管理层**: 统一配置管理系统
3. **代码审查层**: 提交前硬编码检查
4. **CI/CD层**: 自动化硬编码扫描
5. **运行时层**: 配置验证和警告

### **硬编码检查工具**:
```bash
# 检查硬编码配置
npm run check-hardcoded

# 检查环境变量使用
npm run check-env-usage

# 检查API路由硬编码
npm run check-routes
```

### **禁止的硬编码模式**:
```javascript
// ❌ 绝对禁止！
const port = 33037
const apiUrl = 'http://localhost:33037'
const secret = 'my-secret-key'
const dbConfig = { host: 'localhost', port: 5432 }

// ✅ 必须使用配置系统
import config from './config'
const port = config.server.port
const apiUrl = config.api.baseUrl
const secret = config.auth.jwtSecret
const dbConfig = config.database
```

## 🔄 规范化修复流程

### **导入文件的规范化处理**:
```
1. 检查文件 → 2. 识别硬编码 → 3. 提取配置 → 
4. 更新环境变量 → 5. 修改代码 → 6. 验证测试
```

### **旧代码迁移规范**:
1. **逐步迁移**: 不要一次性重写所有代码
2. **兼容性**: 保持API向后兼容
3. **测试覆盖**: 迁移前必须有测试
4. **文档更新**: 同步更新相关文档
5. **回滚计划**: 准备回滚方案

### **新代码开发规范**:
1. **配置优先**: 先定义环境变量和配置
2. **类型安全**: 使用TypeScript类型定义
3. **测试驱动**: 先写测试，再写实现
4. **文档同步**: 代码和文档同步更新
5. **代码审查**: 提交前必须通过审查
