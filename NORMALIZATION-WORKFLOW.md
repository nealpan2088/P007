# 麒麟项目 - 规范化开发工作流

## 🎯 核心工作流原则

**渐进开发 + 规范化修复 + 硬编码防护 = 高质量可维护代码**

## 🔄 完整开发工作流

### **阶段 1: 需求分析**
```
输入: 功能需求
输出: 技术设计文档
工具: 无
检查: 需求完整性
```

### **阶段 2: 技术设计**
```
输入: 技术设计文档
输出: 环境变量定义 + API路由定义
工具: CONFIGURATION-GUIDE.md
检查: 无硬编码设计
```

### **阶段 3: 环境配置**
```
输入: 环境变量定义
输出: .env.development + .env.example
工具: 配置管理系统
检查: 必需环境变量完整
```

### **阶段 4: 代码实现**
```
输入: 技术设计 + 环境配置
输出: 功能代码
工具: 渐进开发规范
检查: 无硬编码实现
```

### **阶段 5: 规范化检查**
```
输入: 功能代码
输出: 检查报告
工具: check-hardcoded.sh
检查: 硬编码、配置、路由
```

### **阶段 6: 测试验证**
```
输入: 功能代码
输出: 测试结果
工具: 单元测试 + 集成测试
检查: 功能正确性
```

### **阶段 7: 代码审查**
```
输入: 功能代码 + 测试结果
输出: 审查意见
工具: Git提交前钩子
检查: 代码质量、规范符合
```

### **阶段 8: 部署测试**
```
输入: 审查通过的代码
输出: 部署结果
工具: 部署脚本
检查: 环境兼容性
```

### **阶段 9: 文档更新**
```
输入: 部署成功的功能
输出: 更新后的文档
工具: 文档系统
检查: 文档完整性
```

## 🛡️ 硬编码防护工作流

### **防护层级 1: 设计阶段防护**
```bash
# 在设计阶段就避免硬编码
1. 先定义环境变量
2. 再设计配置结构
3. 最后写代码实现
```

### **防护层级 2: 开发阶段防护**
```bash
# 开发时实时检查
npm run check:hardcoded    # 硬编码检查
npm run check:config       # 配置验证
npm run check:routes       # 路由检查
```

### **防护层级 3: 提交阶段防护**
```bash
# Git提交前自动检查
git commit -m "feat: 新功能"
# ↓ 自动运行
# 1. 硬编码检查
# 2. 配置验证
# 3. 代码检查
# 4. 测试运行
```

### **防护层级 4: CI/CD阶段防护**
```bash
# 持续集成管道
1. 代码拉取
2. 依赖安装
3. 硬编码检查
4. 配置验证
5. 测试运行
6. 构建打包
7. 部署测试
```

### **防护层级 5: 运行时防护**
```bash
# 应用启动时验证
1. 环境变量加载
2. 配置验证
3. 路由验证
4. 数据库连接
5. 服务启动
```

## 🔧 导入文件规范化工作流

### **场景 1: 导入旧项目代码**
```
1. 运行规范化导入脚本
   ./scripts/normalize-import.sh <文件或目录>

2. 查看生成的报告
   <文件>.normalize-report.txt

3. 根据报告修复硬编码
   - 提取硬编码值到环境变量
   - 更新配置管理系统
   - 修改代码使用配置系统

4. 运行检查验证
   ./scripts/check-hardcoded.sh

5. 测试修复后的功能
   npm test
```

### **场景 2: 导入第三方库/组件**
```
1. 检查库的配置方式
   - 是否支持环境变量？
   - 是否有硬编码配置？

2. 创建包装层（如果需要）
   - 封装硬编码的库
   - 通过包装层提供配置接口

3. 更新环境变量定义
   - 添加库需要的配置
   - 更新 .env.example

4. 更新配置管理系统
   - 添加库配置模块
   - 提供类型定义

5. 测试集成功能
   - 单元测试
   - 集成测试
```

### **场景 3: 复制现有代码模块**
```
1. 不要直接复制粘贴！
2. 分析代码中的硬编码
3. 提取可配置的部分
4. 创建通用配置接口
5. 实现新的模块
6. 运行规范化检查
```

## 🚨 常见问题处理流程

### **问题 1: 发现硬编码**
```
症状: 检查脚本报告硬编码
处理:
1. 识别硬编码的类型（端口、URL、密钥等）
2. 提取到环境变量
3. 更新配置管理系统
4. 修改代码使用配置
5. 重新运行检查
```

### **问题 2: 配置验证失败**
```
症状: config.validate() 失败
处理:
1. 查看错误信息
2. 检查缺失的环境变量
3. 更新 .env.development
4. 验证配置格式
5. 重新运行验证
```

### **问题 3: 路由常量未使用**
```
症状: 代码中使用硬编码路径
处理:
1. 检查路由常量系统
2. 添加缺失的路由定义
3. 导入路由常量
4. 替换硬编码路径
5. 测试API调用
```

### **问题 4: 环境变量缺失**
```
症状: process.env.XXX 为 undefined
处理:
1. 检查 .env.development 文件
2. 添加缺失的环境变量
3. 更新 .env.example
4. 重启开发服务
5. 验证环境变量加载
```

## 🎯 成功的关键习惯

### **习惯 1: 先配置，后编码**
```
❌ 错误: 先写代码，发现需要配置再添加
✅ 正确: 先定义环境变量和配置，再写代码
```

### **习惯 2: 小步提交，频繁检查**
```
❌ 错误: 写很多代码，最后一起提交和检查
✅ 正确: 每完成一个小功能就提交和检查
```

### **习惯 3: 测试驱动，配置先行**
```
❌ 错误: 先实现功能，再写测试和配置
✅ 正确: 先写测试和配置，再实现功能
```

### **习惯 4: 文档同步，知识共享**
```
❌ 错误: 代码写完再补文档
✅ 正确: 代码和文档同步更新
```

### **习惯 5: 定期审查，持续改进**
```
❌ 错误: 只在新功能时关注规范
✅ 正确: 定期审查现有代码，持续改进
```

## 🚀 立即开始规范化开发

### **第一步: 设置开发环境**
```bash
# 1. 复制环境变量
cp apps/backend/.env.example apps/backend/.env.development
cp apps/frontend/.env.example apps/frontend/.env.development

# 2. 启动开发服务
./start-dev.sh

# 3. 运行初始检查
cd apps/backend && npm run check:all
cd apps/frontend && npm run check:all
```

### **第二步: 开始新功能开发**
```bash
# 1. 创建功能分支
git checkout -b feature/your-feature

# 2. 先定义配置（如果需要）
# 编辑 .env.development 和 .env.example

# 3. 更新配置管理系统
# 编辑 src/config/index.js 或 index.ts

# 4. 实现功能代码
# 遵循渐进式开发规范

# 5. 运行检查并提交
npm run check:all
git add .
git commit -m "feat: 你的功能描述"
```

### **第三步: 完成功能开发**
```bash
# 1. 运行完整测试
npm test

# 2. 最终检查
npm run check:all

# 3. 代码审查
# 自审或请同事审查

# 4. 合并到开发分支
git checkout develop
git merge feature/your-feature

# 5. 更新文档
# 更新相关文档和CHANGELOG
```
