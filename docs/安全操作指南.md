# 安全操作指南

## 核心原则

**关键指令必须争得用户同意** - 这是最重要的安全原则。

## 问题回顾

### 我的错误
1. **过度自动化**: 创建了可能删除数据库的脚本
2. **缺乏确认**: 没有明确询问用户是否同意执行
3. **权限越界**: 假设可以执行危险操作

### 正确做法
所有关键操作应该：
1. **先询问**: "潘哥，我需要执行XXX操作，可以吗？"
2. **解释风险**: 说明操作的影响和风险
3. **等待确认**: 得到明确同意后再执行
4. **提供选项**: 提供不同的方案选择

## 安全操作分类

### 🔴 高风险操作（必须双重确认）
这些操作可能导致数据丢失或服务中断：

| 操作 | 风险 | 确认要求 |
|------|------|----------|
| 删除数据库 | 永久数据丢失 | 双重确认 + 备份验证 |
| 重置数据库 | 所有数据清空 | 双重确认 + 备份验证 |
| 修改生产配置 | 服务中断 | 变更窗口 + 回滚计划 |
| 删除文件 | 数据丢失 | 确认文件内容 + 备份 |

### 🟡 中风险操作（需要确认）
这些操作可能影响系统状态：

| 操作 | 风险 | 确认要求 |
|------|------|----------|
| 创建数据库 | 可能覆盖现有数据 | 检查是否存在 + 确认 |
| 运行迁移 | 可能破坏数据结构 | 测试环境验证 + 确认 |
| 安装依赖 | 可能引入安全漏洞 | 版本检查 + 确认 |
| 重启服务 | 短暂服务中断 | 低流量时段 + 确认 |

### 🟢 低风险操作（可以自动化）
这些操作风险较低：

| 操作 | 风险 | 自动化程度 |
|------|------|------------|
| 检查状态 | 无风险 | 完全自动化 |
| 读取日志 | 无风险 | 完全自动化 |
| 生成报告 | 无风险 | 完全自动化 |
| 代码格式化 | 低风险 | 可以自动化 |

## 安全确认机制

### 1. 双重确认机制
```bash
# 第一层：风险警告
echo "⚠️  此操作将删除数据库 '$DB_NAME'，所有数据将永久丢失！"
read -p "确认了解风险吗？(输入 'CONFIRM' 继续): " -r

# 第二层：具体操作确认
if [[ "$REPLY" == "CONFIRM" ]]; then
    echo "🔴 即将删除数据库 '$DB_NAME'"
    read -p "确认要删除吗？输入 'DELETE' 确认: " -r
    if [[ "$REPLY" == "DELETE" ]]; then
        # 执行操作
    fi
fi
```

### 2. 备份验证机制
```bash
# 在执行危险操作前验证备份
if [ -f "/path/to/backup.sql" ]; then
    echo "✅ 备份文件存在"
    read -p "已确认备份，继续执行吗？(y/N): " -n 1 -r
else
    echo "❌ 备份文件不存在"
    echo "请先创建备份再继续"
    exit 1
fi
```

### 3. 环境检查机制
```bash
# 检查当前环境
if [[ "$NODE_ENV" == "production" ]]; then
    echo "🔴 当前是生产环境！"
    echo "请确认："
    echo "1. 是否在变更窗口内？"
    echo "2. 是否有回滚计划？"
    echo "3. 是否已通知相关人员？"
    read -p "确认所有条件满足吗？(输入 'PRODUCTION' 继续): " -r
    if [[ "$REPLY" != "PRODUCTION" ]]; then
        exit 1
    fi
fi
```

## 手动操作指南

如果你不信任自动化脚本，可以手动执行：

### 数据库初始化（手动）
```bash
# 1. 检查当前数据库状态
PGPASSWORD=postgres psql -h localhost -p 5432 -U postgres -l

# 2. 备份现有数据库（如果存在）
PGPASSWORD=postgres pg_dump -h localhost -p 5432 -U postgres qilin_dev > backup-$(date +%Y%m%d).sql

# 3. 创建新数据库
PGPASSWORD=postgres psql -h localhost -p 5432 -U postgres -c "CREATE DATABASE qilin_dev;"

# 4. 运行迁移
cd apps/backend
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/qilin_dev" npx prisma migrate dev

# 5. 验证数据库
PGPASSWORD=postgres psql -h localhost -p 5432 -U postgres -d qilin_dev -c "\dt"
```

### 环境配置（手动）
```bash
# 1. 复制环境文件模板
cp apps/backend/.env.example apps/backend/.env.development
cp apps/frontend/.env.example apps/frontend/.env.development

# 2. 编辑环境文件
# 使用你喜欢的编辑器编辑 .env.development 文件
# 设置 DATABASE_URL、JWT_SECRET 等

# 3. 验证环境变量
cd apps/backend
node -e "console.log('DATABASE_URL:', process.env.DATABASE_URL || '未设置')"
```

## 紧急恢复

### 如果脚本出错
1. **立即停止**: Ctrl+C
2. **检查状态**: 
   ```bash
   # 检查数据库状态
   PGPASSWORD=postgres psql -h localhost -p 5432 -U postgres -l
   
   # 检查服务状态
   ps aux | grep node
   ```
3. **查看日志**:
   ```bash
   tail -f /tmp/backend-log.txt
   tail -f /tmp/frontend-log.txt
   ```
4. **手动恢复**:
   - 如果有备份: 从备份恢复
   - 如果没有备份: 根据错误信息手动修复

### 恢复脚本
```bash
#!/bin/bash
# restore-database.sh - 数据库恢复脚本

echo "=== 数据库恢复 ==="
echo ""

# 1. 检查备份文件
BACKUP_FILE="backup-$(date +%Y%m%d).sql"
if [ ! -f "$BACKUP_FILE" ]; then
    echo "❌ 备份文件不存在: $BACKUP_FILE"
    echo "请手动恢复"
    exit 1
fi

echo "✅ 找到备份文件: $BACKUP_FILE"
read -p "确认从备份恢复数据库吗？(输入 'RESTORE' 确认): " -r
if [[ "$REPLY" != "RESTORE" ]]; then
    echo "操作已取消"
    exit 0
fi

# 2. 恢复数据库
echo "恢复数据库..."
PGPASSWORD=postgres psql -h localhost -p 5432 -U postgres -d qilin_dev < "$BACKUP_FILE"

echo "✅ 数据库恢复完成"
```

## 总结

### 安全原则
1. **用户同意第一**: 所有关键操作必须得到用户明确同意
2. **风险透明**: 清晰说明操作的风险和影响
3. **随时取消**: 提供安全的取消机制
4. **备份优先**: 危险操作前必须有备份

### 我的承诺
作为AI助手，我承诺：

1. **不再自动执行危险操作** - 所有关键操作都会先询问
2. **提供详细的风险说明** - 让你了解操作的影响
3. **提供安全的替代方案** - 包括手动操作指南
4. **尊重你的决策** - 如果你说不，就停止操作

### 你的权利
作为用户，你有权：

1. **随时停止任何操作**
2. **要求详细的风险说明**
3. **选择手动操作替代自动化**
4. **审查和修改所有自动化脚本**

**安全是我们共同的责任，你的同意是执行任何关键操作的前提。**
